# 시각화 스킬

## 목적

대시보드와 시각 요약에서 지금 판단해야 할 내용을 먼저 보이게 만드는 것이 목표입니다.
반드시 `DESIGN.md`, `.claude/rules/input-contract.md`, `00_input_manifest.json` 을 먼저 읽고 시작합니다.

## 작업 순서

1. 데이터 규모 파악
2. 화면의 핵심 메시지 한 줄 정리
3. UI용 짧은 문구 설계
4. 메인 차트 1개 선정
5. 보조 차트 2개 이하 선정
6. 표 상한 반영
7. HTML 대시보드 제작

## 시각화 원칙

- 메인 차트 하나가 화면의 중심을 잡아야 합니다
- 큰 CSV는 집계와 축약이 먼저입니다
- 작은 CSV는 구조 설명과 표가 더 중요할 수 있습니다
- 범주가 많은 차원은 `상위 N개 + 기타`
- 표는 보조 수단이지 원시 데이터 덤프가 아닙니다
- 여러 소스일수록 출처와 관계를 더 짧고 분명하게 적습니다
- 관계가 애매한 파일은 한 차트에 섞지 않습니다

## 디자인 원칙

- 어두운 관제 화면 느낌
- `Space Grotesk` + `Inter` + `IBM Plex Mono`
- 자주빛 바탕 + 라임 포인트
- 화면은 큰 덩어리 몇 개로 읽히게
- 카드가 필요할 때만 카드처럼 보이게

## 문구 원칙

- 라벨은 짧고 강해야 합니다
- 긴 설명은 헤더가 아니라 본문으로 내려야 합니다
- 상단 제목은 6~12자 안팎으로 끊습니다
- 탭 라벨은 2~4자
- 패널 제목은 2~8자
- 액션 제목은 2~6자
- `핵심 질문 1 제목 — ...` 처럼 설명이 붙은 제목은 금지합니다

## 출력 형식

### 1. 화면 한 줄 방향
- 이 화면이 어떤 판단을 돕는지

### 2. UI용 짧은 문구
- 상단 제목:
- 상단 소스 메타:
- 탭 라벨 1:
- 탭 라벨 2:
- 탭 라벨 3:
- 패널 제목 1:
- 패널 제목 2:
- 패널 제목 3:
- 액션 제목 1:
- 액션 제목 2:
- 액션 제목 3:

### 3. 화면 구성
- 상단 헤드
- 요약
- 지표
- 메인
- 보조 영역
- 결정표
- 액션

### 4. 소스 표시 방식
- 패널별 근거 소스:
- 관계 표기 방식:
- 결합 전제 또는 비교 전제:

### 5. 차트 선택 이유
- 차트 종류
- 왜 이 차트인지
- `상위 N개 + 기타` / 집계 규칙

### 6. HTML 대시보드
- `runs/latest/outputs/06_dashboard.html`

## 금지

- 첫 화면이 카드 모자이크처럼 보이는 구성
- 탭당 차트 4개 초과
- 광고 문구 같은 제목
- KPI 카드 5개 초과
- 길고 설명적인 탭/버튼 문구
- 소스가 다른 파일을 근거 없이 한 그래프에 섞기
