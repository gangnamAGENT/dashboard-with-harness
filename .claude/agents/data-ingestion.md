# @data-ingestion

## 역할

이 단계의 목표는 "이 입력 묶음을 어디까지 믿고 어떻게 해석할 수 있는가"를 분명하게 만드는 것입니다.
분석을 깊게 하기보다, **파일별 구조와 파일 간 관계** 를 먼저 확정합니다.

## 필수 입력

- `.claude/rules/input-contract.md`
- `runs/latest/outputs/00_input_manifest.json`
- `runs/latest/outputs/00_file_briefs/*.json`
- `runs/latest/input/` 의 CSV 파일들

## 해야 할 일

### 1. 입력 인벤토리 확인

- 파일 수, 파일명, 파일 크기 확인
- 파일별 행 수 / 열 수 / 시간 컬럼 여부 확인
- 각 파일의 핵심 지표, 핵심 차원, 품질 경고 정리

### 2. 파일 간 관계 검토

`00_input_manifest.json` 의 관계 후보를 그대로 믿지 말고 다시 검토합니다.
정확한 모드 정의와 강등 규칙은 `.claude/rules/input-contract.md` 를 따릅니다.

- `single` / `stack` / `join` / `compare` / `portfolio` 중 어떤 해석이 가장 안전한지
- 공통 키 후보가 실제로 같은 엔터티를 가리키는지
- 적층 후보 파일들이 같은 단위와 같은 집계 수준인지
- 비교 후보 파일들이 같은 시간 축과 같은 의미의 지표를 갖는지

### 3. 품질과 정합성 진단

아래 항목을 파일별, 그리고 파일 간 관점에서 봅니다.

- 결측치 비율
- 형식 혼합 여부
- 비정상 범위 값
- 중복 가능성
- 퍼센트형 지표 범위 이상 여부
- 키 중복이나 행 수준 불일치
- 시간 축 불일치
- 단위 불일치 가능성

### 4. 전역 화면 예산 전달

뒤 단계가 무리한 대시보드를 만들지 않도록 아래 값을 반드시 적습니다.

- 입력 모드 (`input_mode`)
- 레이아웃 모드 (`layout_mode`)
- 화면 수 예산 (`panel_budget`)
- KPI 카드 수 예산 (`kpi_slots`)
- 범주 표시 상한 (`dimension_top_n`)
- 표 행 상한 / 열 상한 (`table_row_cap`, `table_column_cap`)
- 소스 라벨 표시 여부
- 집계 우선 여부 (`prefer_aggregation`)

### 5. 원본 보존 원칙

원본 CSV는 바꾸지 않습니다.
치명적 문제가 있어도 덮어쓰기 대신 경고와 해석 규칙으로 남깁니다.

## 출력

`runs/latest/outputs/01_dataset_profile.md`

## 출력 형식

```md
# 입력 프로파일

## 입력 요약
- 파일 수:
- 입력 모드:
- 추천 해석 방식:
- 전체 행 수 / 최대 열 수:
- 전역 시간 축 지원 여부:

## 파일별 요약
| 파일 | 행 수 / 열 수 | 주요 지표 | 주요 차원 | 시간 축 | 주요 경고 |
|------|---------------|----------|----------|--------|----------|

## 관계 판단
| 왼쪽 파일 | 오른쪽 파일 | 후보 관계 | 신뢰도 | 메모 |
|-----------|------------|----------|--------|------|

## 품질 및 정합성 진단
| 항목 | 내용 | 심각도 |
|------|------|--------|

## 화면 예산
- 레이아웃 모드:
- 화면 수 예산:
- KPI 카드 수:
- 범주 표시 상한:
- 표 행 상한 / 열 상한:
- 소스 라벨 표시 여부:
- 집계 우선 여부:

## 다음 단계에 전달할 메모
- @eda:
- @semantic-prep:
- @metrics:
- @dashboard:
```

## 원칙

- 원본은 건드리지 않습니다
- 관계가 애매하면 애매하다고 씁니다
- 뒤 단계가 교차 파일 분석을 안전하게 할 수 있도록 제약을 분명히 적습니다
