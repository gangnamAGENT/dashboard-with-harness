# @analysis

## 역할

대시보드와 보고서에 실릴 핵심 인사이트를 선별하고 압축합니다.
여러 CSV가 있더라도, 인사이트마다 어떤 파일을 근거로 했는지와 어떤 관계를 가정했는지 남겨야 합니다.

## 필수 입력

- `.claude/rules/input-contract.md`
- `runs/latest/outputs/00_input_manifest.json`
- `runs/latest/outputs/03_problem_definition.md`
- `runs/latest/outputs/04_kpi_summary.md`
- 옵션 A가 실행되었다면 `runs/latest/outputs/01b_dataset_prep.md`
- 원본 CSV들

## 해야 할 일

### 1. 화면 예산 확인

먼저 아래를 읽습니다.

- 화면 수 예산 (`panel_budget`)
- KPI 카드 수 (`kpi_slots`)
- 범주 표시 상한 (`dimension_top_n`)
- 시계열 허용 여부 (`show_time_series`)
- 입력 모드 (`input_mode`)

### 2. 분석 방법 고르기

`00_input_manifest.json` 의 `input_mode`, 관계 후보, 준비 메모를 함께 보고 질문에 맞는 방법만 고릅니다.
정확한 모드 정의와 강등 규칙은 `.claude/rules/input-contract.md` 를 따릅니다.

- 단일 소스면 파일 내부 변화와 분해를 우선합니다
- 비교 모드면 같은 축의 차이를 짧게 정리합니다
- 적층이나 결합은 정합 조건이 확인된 범위에서만 씁니다
- 병렬 해석이 필요한 입력은 파일별 핵심 메시지를 따로 남깁니다

### 3. 전체 → 세부 순서로 해석

1. 전체 현황
2. 주요 분해
3. 원인
4. 이상 신호
5. 기회

### 4. 대시보드용 결과 압축

아래를 반드시 만듭니다.

- 메인 메시지 1개
- UI용 짧은 문구 세트
- 스토리 4문장
- 상단 KPI 카드 설명
- 메인 차트 설명 1개
- 보조 차트 설명 2개 이하
- 액션 3개

## 출력

`runs/latest/outputs/05_analysis_report.md`

## 출력 형식

```md
# 핵심 분석 보고서

## 분석 흐름 한 줄 요약
- ...

## 방법 선택 이유
| 질문 | 방법 | 근거 소스 | 이유 |
|------|------|-----------|------|

## 핵심 인사이트
### 인사이트 1
- 무엇이 보였는지:
- 근거 소스:
- 관계 가정:
- 숫자 근거:
- 왜 중요한지:
- 한계:

### 인사이트 2
- ...

### 인사이트 3
- ...

## 대시보드용 핵심 문장
"..."

## UI용 짧은 문구
- 상단 제목:
- 탭 라벨 1:
- 탭 라벨 2:
- 탭 라벨 3:
- 패널 제목 1:
- 패널 제목 2:
- 패널 제목 3:
- 액션 제목 1:
- 액션 제목 2:
- 액션 제목 3:

## 스토리 4문장 초안
- 현황:
- 원인:
- 영향:
- 기회:

## 메인 시각화 제안
- 어떤 차트인지:
- 근거 소스:
- 왜 이 차트가 중심인지:

## 보조 시각화 제안
- 차트 A:
- 차트 B:

## 액션
- P1:
- P2:
- P3:
```

## 원칙

- 분석량보다 전달력을 우선합니다
- 화면이 감당할 수 없는 인사이트는 버립니다
- 큰 입력 묶음은 집계로 말하고, 작은 입력은 맥락으로 말합니다
- 짧은 문구는 헤더용, 긴 설명은 본문용으로 나눕니다
- 근거 소스와 관계 가정이 없는 인사이트는 올리지 않습니다
