# @problem

## 역할

탐색에서 보인 패턴을 "우리가 무엇을 판단해야 하는가"라는 문제로 바꿉니다.
좋은 문제 정의는 대시보드의 제목과 액션까지 자연스럽게 이어져야 하고,
여러 CSV가 있을 때도 **어떤 소스를 어디까지 묶어 볼지** 가 함께 드러나야 합니다.

## 필수 입력

- `.claude/rules/input-contract.md`
- `runs/latest/outputs/00_input_manifest.json`
- `runs/latest/outputs/02_eda_report.md`
- 옵션 A가 실행되었다면 `runs/latest/outputs/01b_dataset_prep.md`

## 해야 할 일

### 1. 분석 모드 확정

- `00_input_manifest.json` 의 `input_mode` 를 기본으로 씁니다
- 문제 정의에 필요한 해석 범위를 다시 확인하고, 애매하면 더 보수적인 모드로 강등합니다
- 정확한 모드 정의와 강등 규칙은 `.claude/rules/input-contract.md` 를 따릅니다

### 2. 메인 문제 고르기

- 핵심 패턴 중 실제 임팩트가 큰 문제를 고릅니다
- 모호한 표현 대신 수치와 기간이 들어간 문장으로 씁니다
- 교차 파일 문제라면 어떤 소스가 근거인지 적습니다

### 3. 원인 구조 정리

메인 문제를 원인별로 나눕니다.
서로 겹치지 않고, 각각 다른 대응 방향이 나와야 합니다.

### 4. 질문 확정

대시보드와 보고서가 결국 답해야 할 질문 Top 3를 확정합니다.
질문마다 근거 소스와 허용된 관계를 함께 적습니다.

### 5. 화면용 이름 압축

- 확정 질문마다 2~4자짜리 탭 라벨 후보를 만듭니다
- 각 질문에 대응되는 패널 제목 후보를 2~8자로 압축합니다
- 길고 설명적인 제목은 보조 문장으로 내립니다

## 출력

`runs/latest/outputs/03_problem_definition.md`

## 출력 형식

```md
# 문제 정의 및 질문 설계

## 분석 모드
- 확정 모드:
- 왜 이 모드가 맞는지:
- 금지해야 할 해석:

## 메인 문제
- 문제 문장:
- 왜 중요한가:
- 이번 분석의 목표:
- 근거 소스:

## 원인 구조
### 원인 1
- 더 근본적인 이유:
- 대응 방향:
- 확인할 지표:

### 원인 2
- ...

### 원인 3
- ...

## 확정 질문 Top 3
### 질문 1
- 근거 소스:
- 허용된 관계:
- 어떻게 검증할지:
- 어떤 결정을 돕는지:
- 성공 기준:

### 질문 2
- ...

### 질문 3
- ...

## 화면 라벨 후보
- 탭 라벨 1:
- 탭 라벨 2:
- 탭 라벨 3:
- 패널 제목 1:
- 패널 제목 2:
- 패널 제목 3:

## 다음 단계에 전달할 메모
- @metrics 에 꼭 필요한 지표
- @analysis 에서 버리지 말아야 할 관점
- @dashboard 에서 분명히 보여야 할 소스 구분
```

## 원칙

- 설명이 아니라 판단으로 이어지는 문제를 씁니다
- 답할 수 없는 질문은 만들지 않습니다
- 질문은 많지 않아야 합니다
- 뒤 단계가 바로 쓸 수 있게 이름도 짧게 남깁니다
- 안전하지 않은 결합을 전제로 문제를 만들지 않습니다
