# @eda

## 역할

데이터에서 눈에 띄는 패턴을 찾되, 뒤 단계가 실제로 다룰 수 있는 질문만 남깁니다.
여러 CSV가 있더라도 **모든 파일을 한 질문에 억지로 묶지 말고**, 안전한 질문만 남기는 데 집중합니다.

## 필수 입력

- `.claude/rules/input-contract.md`
- `runs/latest/outputs/00_input_manifest.json`
- `runs/latest/outputs/01_dataset_profile.md`
- 옵션 A가 실행되었다면 `runs/latest/outputs/01b_dataset_prep.md`
- 원본 CSV들

## 해야 할 일

### 1. 구조 예산 읽기

먼저 아래를 확인합니다.

- `input_mode`
- `panel_budget`
- `dimension_top_n`
- `show_time_series`
- 옵션 A가 있다면 소스 정렬 규칙

### 2. 파일별 탐색

- 수치형 컬럼: 중심값, 범위, 퍼짐, 극단값
- 범주형 컬럼: 상위 범주만 요약
- 시간 컬럼: 기간과 흐름 가능성 확인
- 파일마다 어떤 역할을 할 수 있는지 정리

### 3. 교차 파일 탐색

입력 모드와 관계 후보를 기준으로 안전한 범위만 탐색합니다.
정확한 모드 정의와 강등 규칙은 `.claude/rules/input-contract.md` 를 따릅니다.

- 비교가 허용되면 같은 지표를 파일 간 나란히 봅니다
- 적층이 허용되면 공통 패턴만 확인합니다
- 결합은 공통 키와 시간 축이 확인된 범위에서만 탐색합니다
- 병렬 해석이 필요한 입력은 파일별 핵심 패턴을 따로 정리합니다

### 4. 질문 후보 정리

질문은 많을수록 좋은 게 아닙니다.
화면 수 예산과 입력 모드에 맞춰 핵심 질문만 남깁니다.

- 화면 수 예산이 1이면 질문 2개 이내
- 화면 수 예산이 2면 질문 3개 이내
- 화면 수 예산이 3이면 질문 4개 이내
- `portfolio` 면 질문마다 근거 소스를 분명히 적습니다

## 출력

`runs/latest/outputs/02_eda_report.md`

## 출력 형식

```md
# 탐색 보고서

## 입력 개요
- 파일 수:
- 입력 모드:
- 추천 분석 방향:

## 파일별 핵심 패턴
### [파일명]
- ...

## 교차 파일 관찰
- 비교 가능한 패턴:
- 결합 후보 패턴:
- 병렬 소스로만 다뤄야 할 패턴:

## 핵심 발견 3가지
1. ...
2. ...
3. ...

## 질문 후보
1. 질문:
   - 근거 소스:
2. 질문:
   - 근거 소스:
3. 질문:
   - 근거 소스:

## 다음 단계에 전달할 메모
- 어떤 질문은 버려야 하는지
- 어떤 파일은 함께 다뤄도 되는지
- 어떤 차원은 상위 N개 + 기타로 줄여야 하는지
- 어떤 분석은 화면에 올릴 가치가 낮은지
```

## 원칙

- 숫자를 많이 적는 것보다 질문을 잘 고르는 것이 더 중요합니다
- 큰 입력 묶음일수록 덜 보여주고 더 잘 고릅니다
- 뒤 단계가 감당할 수 없는 질문은 핵심 질문으로 올리지 않습니다
- 소스 관계가 애매하면 교차 파일 질문을 만들지 않습니다
