# 디자인 기준

## 디자인 방향

이 대시보드는 발표 자료처럼 보이면 안 됩니다.  
운영자가 바로 읽고 판단할 수 있는 **미니멀한 운영 화면** 이 나와야 합니다.

기본 무드는 Sentry 계열의 어두운 제어 화면에서 가져옵니다.

- 짙은 자주빛 배경
- 유리처럼 겹쳐지는 반투명 표면
- 보랏빛 강조색
- 중요한 신호를 찍어 주는 라임 계열 포인트
- 굵고 단단한 제목 글꼴 + 정돈된 본문 + 모노 메타 정보

이 화면은 화려하기보다 **침착하고 정밀해 보여야** 합니다.

## 전체 인상

- 분위기: 어둡고 차분한 분석 화면
- 밀도: 촘촘하지만 숨 막히지 않게
- 우선순위: 메인 차트 1개, 보조 영역 1~2개, 짧은 표 1개
- 피할 것: 베이지 컨설팅 슬라이드, 카드만 잔뜩 늘어선 화면, 쓸데없이 밝은 장식

## 다중 소스 기본 태도

- 파일이 여러 개라고 패널을 파일 수만큼 늘리지 않습니다
- 먼저 `무엇을 함께 볼 수 있는지` 를 정하고, 그다음 화면을 만듭니다
- 한 패널에는 하나의 해석 관계만 둡니다
  - 파일 내부 분석
  - 파일 간 비교
  - 검증된 결합
  - 병렬 소스 요약
- 관계가 애매한 파일은 한 차트에 섞지 않고 소스 구분을 유지합니다

## 글꼴

- 제목 / 구역 제목: `Space Grotesk`
- 본문 / 라벨: `Inter`
- 수치 / 메타 / 코드형 정보: `IBM Plex Mono`

### 글꼴 사용 규칙

- 메인 제목은 짧고 강하게
- 구역 라벨은 모노 글꼴로 작고 또렷하게
- KPI 수치는 크고 단단하게
- 메타 정보는 모노 글꼴로 차분하게
- 한글 대시보드에 장식용 세리프 글꼴은 쓰지 않습니다

## 문구 압축 규칙

- 페이지 키커는 1단어로 끝냅니다
- 상단 제목은 짧은 명사형 또는 신호형으로 씁니다
- 탭 라벨은 2~4자 안팎으로 맞춥니다
- 패널 제목은 2~8자 안팎으로 맞춥니다
- 섹션 제목은 `요약`, `지표`, `메인`, `액션` 처럼 한 단어를 우선합니다
- 버튼과 액션 제목은 2~6자 안팎으로 끊습니다
- 설명은 제목이 아니라 바로 아래 보조 문장으로 내립니다
- `핵심 질문 1 제목 — ...` 같은 설명형 제목은 쓰지 않습니다

### 좋은 예

- `운영`
- `핵심`
- `비교`
- `구조`
- `핵심 신호`
- `세부 비교`
- `데이터 구조`
- `결정표`
- `시나리오`

### 피할 예

- `의사결정자가 바로 읽을 수 있는 운영형 분석 화면`
- `핵심 질문 1 제목 — 메인 차트 하나로 바로 읽히는 결론`
- `즉시 실행 액션`

## 화면 구조

### 바깥 구조

- 화면 전체를 넓게 쓰되, 콘텐츠는 `1440px` 안팎에서 정리
- 바탕은 어둡고, 빛 번짐은 아주 약하게만
- 박스 안에 박스를 계속 넣지 말고 큰 덩어리 몇 개로 위계를 만듭니다

### 패널 기본 구조

1. 상단 헤드
2. 요약
3. 지표
4. 메인: 큰 차트 + 짧은 해석
5. 보조 영역 1~2개
6. 결정표
7. 액션

### 멀티소스 메타 규칙

- 상단 메타에는 파일명 하나를 길게 쓰지 말고 소스 수나 대표 소스만 짧게 적습니다
- 패널마다 근거 소스를 짧게 표시합니다
- `compare`, `join`, `stack`, `portfolio` 중 어떤 해석인지 메타나 보조 문장에 남깁니다
- 결합 전제가 있으면 차트 옆이나 표 위에 짧게 적습니다

### 표면 규칙

- 카드처럼 보이는 박스는 꼭 필요할 때만 씁니다
- 작은 박스를 많이 두기보다, 큰 덩어리 몇 개로 읽히게 만듭니다
- 메인 차트가 화면의 중심을 잡아야 합니다

## 움직임

- 패널 전환: 빠른 페이드 + 아주 약한 상승
- hover: 테두리만 살짝 밝아지고 1~2px 정도 뜨는 느낌
- 슬라이더: 강조색 채움이 바로 바뀌는 정도면 충분
- 배경에서 계속 움직이는 장식 애니메이션은 쓰지 않습니다

## 차트 방향

- 순수 SVG 사용
- 우선순위: 정렬 바 차트, 라인 차트, 히트 스트립, 작은 산점도, 워터폴
- 피할 것: 파이, 게이지, 도넛 여러 개, 장식용 3D
- 한 패널에서 주인공 차트는 하나만 둡니다

### 차트 색 규칙

- 기본 계열: 보라 / 라임 / 옅은 보랏빛
- 리스크: 분홍 / 붉은 계열
- 경고선: 코랄 / 앰버
- 밝은 색은 정말 중요한 정보에만 씁니다

## CSV 규모별 대응

### 아주 작은 CSV

조건:
- 행 수 25 미만

화면 원칙:
- 패널 1개
- KPI 카드 2~3개
- 차트는 많아도 1개
- 구조 설명과 표가 더 중요함
- 억지로 풍성해 보이게 만들지 않음

### 작은 CSV

조건:
- 행 수 25~499

화면 원칙:
- 패널 2개 이내
- KPI 카드 3~4개
- 메인 차트 1개 + 보조 차트 2개 이하
- 원시 표를 보여줄 수는 있지만 행 수는 제한

### 여러 CSV

조건:
- 파일 수 2개 이상

화면 원칙:
- 패널 수보다 소스 관계가 먼저입니다
- 파일 수가 늘어도 탭은 3개를 넘기지 않습니다
- 소스 라벨 또는 소스 배지를 반드시 남깁니다
- 한 차트에 섞는 파일은 관계가 검증된 경우만 허용합니다
- 관계가 애매하면 병렬 패널이나 비교 표로 돌립니다

### 중간 이상 CSV

조건:
- 행 수 500 이상

화면 원칙:
- 패널 2~3개
- 집계를 먼저 함
- 상위 범주만 보여줌
- 집계 기준을 화면에 적음
- 원시 표는 샘플 또는 집계 표로 바꿈

### 열이 많은 CSV

조건:
- 열 수 18 초과

화면 원칙:
- 모든 열을 다 보여주려 하지 않음
- 결과 / 원인 / 세그먼트 중 하나의 렌즈를 먼저 고름
- 표에는 선택된 열만 남김
- 필요하면 컬럼 개요 카드를 따로 둠

### 범주가 너무 많은 차원

조건:
- 차원 안의 범주 수가 대략 12~15개 이상

화면 원칙:
- 상위 N개 + `기타`
- 범주를 전부 늘어놓지 않음

## 문구 톤

- 광고 문구가 아니라 운영 문구를 씁니다
- 제목은 먼저 짧은 명사형이나 신호형으로 씁니다
- 질문형 제목이 필요하더라도 탭과 버튼에는 쓰지 않습니다
- 좋은 예: `핵심 신호`
- 좋은 예: `세부 비교`
- 좋은 예: `데이터 구조`
- 피할 예: `데이터의 힘을 깨워 보세요`

## 절대 규칙

- KPI 카드는 5개를 넘기지 않습니다
- 분석 탭은 3개를 넘기지 않습니다
- 메인 영역 옆 보조 카드는 2개를 넘기지 않습니다
- 표는 가로 스크롤 없이 잘릴 수 없게 만듭니다
- 범주 라벨이 8개를 넘으면 나머지는 `기타` 로 묶습니다
- 믿을 만한 시간 축이 없으면 예측 패널을 만들지 않습니다
- 변수와 반응 지표가 분명하지 않으면 시뮬레이션 위젯을 넣지 않습니다
- 제목과 버튼에 설명형 문장을 넣지 않습니다
- 근거 소스가 없는 교차 파일 차트는 만들지 않습니다

## 기본값

- 탭 수: 1~3
- KPI 카드 수: 3~5
- 표 행 수: 8~12
- 표 열 수: 5~8
- 범주 막대 수: 5~8
- 스토리 칸 수: 4
